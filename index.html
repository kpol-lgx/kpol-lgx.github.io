Indent a code block

源地址：https://vim.fandom.com/wiki/Indent_a_code_block

  • =i{ 该操作在vim普通模式下执行，可以将光标所在的代码块的内部自动处理为正常的缩进，但是不会处理包围代码块的花括号。具体来说，vim将花括号所在的那两行视为一部分，将花括号包围的部分（不含花括号）视为另一部分，这里的处理花括号是指处理花括号所在的那两行，不处理是指不处理花括号，只处理内部文本。所以如果写 if 语句偷懒少了那个回车，之后处理反而方便了。 （Q: 那要是花括号和文本在同一行呢？A:怎么这么多问题。。。把花括号在那一行当作文本部分即可）
  • =a{ 和上一个一样，区别在于这个会处理花括号
  • =2a{ 处理光标所在的块和包围光标的块，花括号都会被处理，这个数字2还可以继续增加

以上三个操作都需要使用{ ，但是这个字符不怎么好打，所以可以用} 和B 这两者之一去替换{ 。（但是看起来都不怎么方便。。。。）

以上是自动处理，如果需要精细控制，那么：

  • >i{ vim普通模式下的操作，光标所在块内部全体向右缩进一个单位，具体缩进多少取决于你的设置。这个不会处理花括号。
  • <i{ 和上一个相对应，这个是向左缩进一个单位。
  • . 以上操作配合 . （重复上一次命令）可以发挥出强大的威力。

如果你的光标非常巧地落在了 { 或者 }上， 那么：

  • =% 自动处理匹配的整个代码块（花括号及内部），调整为合适的缩进
  • >% 或者 <% 向右或者向左缩进整个块，

暂存已修改的文件

这是git add的另一个功能

git add filename

将这个命令理解为“精确地将内容添加的下一次提交中”。注意暂存的只是文件的这一个对应版本，之后再修改，那么git status又会显示这个文件没有提交，其实是指修改后的新版本没有提交。

简化输出

git status的消息总是非常长，加上 -s 或者 --short 可以获得简化输出。

M 表示文件已修改

A 表示文件新添加到暂存区（git add）

状态栏共两列，左栏指明暂存区状态，右栏指明工作区状态。

忽略文件

创建一个名为 .gitignore 的文件，其中的内容可以是正则表达式，git会自动找到匹配的文件并且将其忽略，注意一个或者一类文件占一行。

.gitignore文件格式规范

  • 所有空行以及以#号开头的行都会被git忽略（也就是文件注释）
  • 可以使用标准的glob模式匹配，这将递归地应用在整个工作区中
  • 匹配模式以（/）开始防止递归
  • 匹配模式可以以（/）结尾指定目录
  • 要忽略指定模式以外的文件，可以在模式前加上叹号（!）取反

glob模式是指shell使用的简化了的正则表达式。

2.2-1 \(\Theta (n^3)\)

2.2-2

该算法维持的循环不变式是，在每次交换前，前i个数必定已经排好序，同时，交换之后前i+1个数字也必定排好序，并且终止循环时得到已经排好序的序列。

由于每次寻找的都是A[i..A.length]中的最小值，因此当前n-1个最小值都被找到之后，剩下的最后一个必然就是最大值。

时间复杂度：每次找最小值必定需要遍历剩下的所有未排序的值，因此不论何种情况下，时间复杂度均为 \(\Theta(\frac{n(n+1)}{2})\) 即 \(\Theta(n^2)\)

2.2-3

既然等可能地为数组中任意元素，那么，将数组一分为二，出现在左或右的概率是相等的，因此平均来看，可以认为出现在中间，也就是说，每次需要查找的元素数大约为总体的一半。最坏情况则是每次都需要查找全部的元素才能确定。

平均情况：\(\Theta(\frac{n}{2})\) 也即\(\Theta(n)\)

最坏情况：\(\Theta(n)\)

2.2-4

最好情况通常是输入就是答案。优化最坏运行时间更复杂，通常要减少循环层数。

2.3.1 分治法

递归算法：为了解决一个给定的问题，算法一次或多次递归调用自身来解决紧密相关的若干子问题。

分治：将原问题分解为几个规模较小但类似于原问题的子问题，递归求解这些子问题，然后合并这些解来建立原问题的解。

  • 分解
  • 递归求解（规模足够小时，直接求解）
  • 合并（精髓所在）

比如归并算法：

先将序列分为相等数量的两半，使用归并排序递归排序左右子列，最后将两个已排序的子序列合并为一个序列。

序列长度为1时，认为已经排序好，不必处理。

因为循环次数限制，哨兵不能被复制回数组。合并操作的复杂度是\(\Theta(n)\)
