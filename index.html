Killing a process
shell使用一种被称为signal的Unix交流机制，用来进行进程之间的信息交流。当一个进程受到一个signal时（不论signal是什么），它会先停止正在执行的动作，然后处理信号并根据发过来的signal中的信息决定要不要改变当前工作流的状态。因此signal就是software interrupts。

General recommended practies for AI

发表于 2021-09-21 更新于 2021-09-30

推荐训练

使用以人类为中心的设计方法

  • 设计内置功能时应当适当公开：清晰和可控对于良好的用户体验非常重要
  • 考虑增加和帮助：如果一个答案的概率很高并且能满足大量不同的使用者和案例，那么只给一个答案是合适的。在其他情况下，
    系统提供多个选项给用户可能是最好的选择。技术上，从一个答案上要获得较高的精度肯定比多个答案要困难许多。
  • 在设计过程的早期对潜在的不利反馈进行建模，然后在全面部署之前对一小部分流量进行实时测试和迭代。
  • 与一组不同的用户和用户方案进行互动，并在项目开发之前和整个过程中纳入反馈。这将为项目构建丰富的用户视角，并增加受益于该技术的人数

本地操作
安装 git 和 Node.js

使用npm安装hexo 实际是hexo-cli

初始化

folder-name 改成自己的文件夹名称。npm install后面没有任何其他内容。

从本地预览，初始是一个Hello World!以及官方自带的主题

github操作
github上新建仓库
要求 ： 必须以 username.github.io 命名
其中 username 必须是自己的github账户名

本地操作
在前面的 $ hexo init 命令所创建的文件夹（暂时称为站点文件夹）中找到_config.yml，修改deploy下的选项

注意：冒号后面必须有一个空格，repo还有其他写法，但不再介绍，后两个可选

之后

github操作
前往github，到所建的仓库下，在仓库的设置中把默认分支改成master,因为_config.yml里的默认分支就是master。愿意折腾的可以自己改成其他分支。

其他
如果想要更改主题，请到hexo官网找到主题下载（其实是从github上下载），下载到theme本地文件夹（这个文件夹就在上面说的站点文件夹里），之后去_config.yml里改对应设置（这里自行查看主题自带的README文档）。最好执行一下 $ hexo d

注意：本地更改过后有可能无法在网页上显示改变，可能需要等待一会。如果长时间没有反应，请执行$ hexo clean && hexo generate && hexo deploy ，如果依然没有反应，这边建议直接在本地重新开一个。

更多内容请参考官方文档
