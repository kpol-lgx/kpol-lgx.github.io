Job Control

大多数时候，使用Ctrl-c可以中断大多数任务。那么它如何工作的？如果失败了应该采用什么方式？

Killing a process

shell使用一种被称为 signal 的Unix交流机制，用来进行进程之间的信息交流。当一个进程受到一个 signal 时（不论 signal 是什么），它会先停止正在执行的动作，然后处理信号并根据发过来的 signal 中的信息决定要不要改变当前工作流的状态。因此 signal 就是software interrupts。

keywords signal
Ctrl-c SIGINT
Ctrl-\ SIGQUIT
Ctrl-z SIGSTOP
kill SIGTERM

kill command

SIGSTP是SIGSTOP的终端简写版本

Pausing and backgrounding processes

SIGSTOP会让进程暂停。

fg,bg用来恢复暂停的工作，分别对应前台和后台。

jobs列出当前窗口下所有的未完成任务。这些任务可以使用PID引用（尝试pgrep搜索）

pgrep

可以通过百分号紧跟工作编号的方式引用进程。$! 可以用来引用最后一个进程。

Ctrl-z让正在运行的程序暂停，紧接着输入bg可以让任务在后台继续执行。

不过，后台进程实质上是终端的子进程，如果关闭终端（将发送一个SIGHUP信号），后台子进程也将同时终止。为了防止这种情况发生，使用nohup（这是一个用来忽略SIGHUP的封装）来运行程序。如果进程已经开始，那么使用disown

有一个比较特殊的 signal ，SIGKILL ，因为其不能被进程捕获，并且总是将进程立刻停止。不过这样会有一些负面效果，比如留下孤立的子进程。

了解更多关于 signal 可以在终端键入man signal 或者 kill -l

General recommended practies for AI

发表于 2021-09-21 更新于 2021-09-30

推荐训练

使用以人类为中心的设计方法

  • 设计内置功能时应当适当公开：清晰和可控对于良好的用户体验非常重要
  • 考虑增加和帮助：如果一个答案的概率很高并且能满足大量不同的使用者和案例，那么只给一个答案是合适的。在其他情况下， 系统提供多个选项给用户可能是最好的选择。技术上，从一个答案上要获得较高的精度肯定比多个答案要困难许多。
  • 在设计过程的早期对潜在的不利反馈进行建模，然后在全面部署之前对一小部分流量进行实时测试和迭代。
  • 与一组不同的用户和用户方案进行互动，并在项目开发之前和整个过程中纳入反馈。这将为项目构建丰富的用户视角，并增加受益于该技术的人数

本地操作 安装 git 和 Node.js

使用npm安装hexo 实际是hexo-cli

初始化

folder-name 改成自己的文件夹名称。npm install后面没有任何其他内容。

从本地预览，初始是一个Hello World!以及官方自带的主题

github操作 github上新建仓库 要求 ： 必须以 username.github.io 命名 其中 username 必须是自己的github账户名

本地操作 在前面的 $ hexo init 命令所创建的文件夹（暂时称为站点文件夹）中找到_config.yml，修改deploy下的选项

注意：冒号后面必须有一个空格，repo还有其他写法，但不再介绍，后两个可选

之后

github操作 前往github，到所建的仓库下，在仓库的设置中把默认分支改成master,因为_config.yml里的默认分支就是master。愿意折腾的可以自己改成其他分支。

其他 如果想要更改主题，请到hexo官网找到主题下载（其实是从github上下载），下载到theme本地文件夹（这个文件夹就在上面说的站点文件夹里），之后去_config.yml里改对应设置（这里自行查看主题自带的README文档）。最好执行一下 $ hexo d

注意：本地更改过后有可能无法在网页上显示改变，可能需要等待一会。如果长时间没有反应，请执行$ hexo clean && hexo generate && hexo deploy ，如果依然没有反应，这边建议直接在本地重新开一个。

更多内容请参考官方文档

修改站点 _config.yml以及主题下 _config.yml 均不生效，hexo clean && hexo g 同样无用。最终 npm uninstall hexo-theme-keep 卸载了原先的主题，更换为next主题。事后思考，大概是原先的主题没有更新。

数学公式显示错误，按照官方文档启用mathjax，更换渲染器为hexo-renderer-pandoc 。支持数学公式的渲染器有的是由个人维护的，因此更新看作者心情，旧版本很可能会WARN ，但只要不影响使用，一切好说。盲目更新可能导致兼容问题，老老实实用稳定版本就好。npm官网上下载最多的包版本一般最稳定。

更改之后还是无法渲染，最后发现是站点目录下的 _config.next.yml 文件没有修改。（我。。。。。）

总结一下：hexo官方自带的渲染器不支持数学公式，按照next主题文档建议，启用mathjax，更换渲染器。另外，hexo的配置文件优先级是 **站点目录下的 _config.yml > 站点目录下的 _config.theme.yml > 主题目录下的 _config.yml** 。主题目录升级会更改其目录下的_config.yml，所以按照next主题官方建议，修改站点目录下的 _config.next.yml 文件最好。

每页都渲染数学公式会降低加载速度，只在需要的post内的front matter 加上 mathjax: true 即可。

\\ 数学公式换行，但是 html 不认这个，直接转义，懒得改了，用逗号吧。
